AdversaryGraph · post-v6 current development

Unified Intelligence RAG and MCP

AdversaryGraph builds one provenance-preserving retrieval corpus over selected ATT&CK, actor, IOC, CVE, report, knowledge, Threat Radar, Threat Hunting, Evidence Graph, and sanitized asset records. It returns source-linked evidence for analyst review and can propose—but never automatically apply—ATT&CK Navigator techniques.

Release and deployment boundary: this capability is post-v6 current development. It must not be attributed to the immutable AdversaryGraph v6.0.0 release. This local presentation copy is not the deployed Docusaurus route; use the linked source guide until that documentation deployment is published.

How does the retrieval architecture work?

Allowlisted collectors create canonical RAG documents and bounded chunks from authoritative platform tables. Retrieval combines exact identifier matching, PostgreSQL full-text search, optional pgvector cosine search, reciprocal-rank fusion, business-profile reranking, and a bounded one-hop relationship expansion.

authoritative records
→ field-allowlisted documents and chunks
→ exact + lexical + optional vector retrieval
→ business-context reranking and bounded relationship expansion
→ routes, TLP, freshness, scores, warnings, and citations
→ governed AI synthesis or an expiring Navigator proposal
→ explicit analyst review and confirmation

Vectors remain in PostgreSQL with their source records. A vector is a retrieval aid; it is not evidence confidence, attribution probability, or proof of relevance.

Which records can Unified RAG search?

“Unified” means one controlled search contract over twelve implemented source types. It does not mean every database table or column is embedded.

How is sensitive intelligence governed?

  • Only curated fields are indexed; raw feed/provider payloads, credentials, audit tables, and arbitrary files are excluded.
  • Every chunk retains source type, stable source ID, canonical route, TLP, content hash, source version, and index time.
  • Client profiles are private reranking/generation context and are not globally retrievable corpus documents.
  • TLP:AMBER+STRICT, TLP:RED, and legally sensitive results cannot be sent to a cloud model.
  • Retrieved text is untrusted data. Instructions inside reports do not become system instructions or tool calls.
  • Generated answers must use verified source markers; unknown citations fail the request.
  • Proposed ATT&CK IDs are checked against the current local catalog, and stale chunk hashes reject an answer.

Field allowlisting is not a general DLP guarantee. Operators remain responsible for removing unnecessary secrets and applying correct TLP/legal markings before ingestion.

What can an analyst ask?

Business-context IOC discovery

An analyst can ask for intelligence relevant to a business profile such as an Israel-based technology company. The profile can improve ranking, but the result remains source-bound and is not a targeting or attribution claim.

{
  "query": "Israel technology sector phishing infrastructure",
  "source_types": ["actor_intel", "ioc", "attack_group", "attack_campaign", "analysis_report", "attack_technique"],
  "domain": "enterprise-attack",
  "limit": 12
}

Navigator technique proposal

The assistant may return an expiring, checksum-bound proposal containing only cited, locally valid techniques. Confirmation records a receipt; it does not automatically save a layer or mutate the Navigator workspace.

Which REST operations are exposed?

  • GET /api/rag/status — corpus, embedding, coverage, and freshness status.
  • POST /api/rag/search — exact, lexical, and optional vector retrieval.
  • GET /api/rag/entity/{type}/{id} — one sanitized indexed entity.
  • POST /api/rag/assist — citation-bound answer and optional proposal.
  • POST /api/rag/proposals/{id}/confirm — confirm a proposal without saving a layer.
  • POST /api/rag/reindex and GET /api/rag/index-runs — governed corpus administration.

Analysis calls require run_analysis; profile administration requires manage_intel; index administration requires manage_feeds.

What does the MCP server allow?

The local stdio MCP surface reuses the authenticated REST boundary. It exposes exactly four advisory tools: search_intelligence, ask_intelligence, get_indexed_entity, and propose_navigator_layer.

It cannot run arbitrary SQL, fetch arbitrary URLs, configure feeds, read secrets or raw provider JSON, execute simulations, forward SIEM data, confirm proposals, or mutate Navigator state.

What are the known limits?

  • Indexed coverage is not proof that no relevant intelligence exists elsewhere.
  • Semantic similarity does not prove operational relevance; analysts must review relationships and evidence.
  • Relationship expansion is bounded and non-recursive, not a complete knowledge-graph traversal.
  • Source freshness, attribution, CVE-to-TTP links, and sector labels inherit source-record limitations.
  • The current single-workspace model is not a hard tenant-isolation boundary.
  • MCP integration is not permission to automate blocking, exploitation, hunt execution, response, or Navigator mutation.
