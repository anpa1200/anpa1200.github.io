## Executive summary

The site is materially ahead of the original brief. Internal search is already generated from rendered HTML with Pagefind 1.5.2, exact ATT&CK identifiers and aliases are boosted, a no-JavaScript fallback exists, canonical coverage is complete for indexable pages, and `robots.txt`, an RSS feed, `llms.txt`, and `llms-full.txt` already exist. The current GitHub Pages deployment and its latest scheduled validation completed successfully.

The remaining work is a consistency and validation overhaul, not a search rewrite. The highest-priority findings are:

1. The production search build found 42 URLs advertised by federated sitemaps that no longer resolve to indexable pages.
2. The stated `ai-train=no` policy conflicts with explicit `Allow: /` rules for training crawlers.
3. JSON-LD is widespread and syntactically valid, but it is not a connected sitewide entity graph and has not been proven error-free in Google and Schema.org validators.
4. Sitemap and feed artifacts are checked in rather than regenerated by the deployment workflow; sitemap `lastmod` values use file modification time rather than an authoritative content date.
5. A small number of structural defects remain, while image dimension and semantic-landmark coverage need a broader deterministic pass.

The audit itself made no production changes. The subsequently approved
implementation updates source, generated artifacts, and the Pages workflow.

## Decisions required before implementation

### 1. Current build and deployment setup

The site uses plain static HTML plus generated documentation/project snapshots. GitHub Actions stages the publish artifact and runs Node 22 post-processing. Pagefind is already installed as a development dependency and is built against the staged site, including approved remote sitemap content.

Recommended decision: retain this architecture. No new build dependency is required for the proposed first implementation pass. Any change to `.github/workflows/pages.yml` will be submitted separately for approval, as requested.

### 2. AI training stance

The current intent appears to be **allow search/retrieval but block training** because `robots.txt` declares:

```text
Content-Signal: search=yes, ai-input=yes, ai-train=no
```

However, the same file explicitly allows GPTBot, ClaudeBot, Google-Extended, CCBot, Bytespider, and other crawlers. The policy is therefore internally inconsistent.

Recommended decision: allow user-triggered and search/retrieval agents, block recognized training crawlers, and document that `robots.txt` is advisory and cannot control non-compliant bots or ordinary browser signatures. Confirm this stance before the crawler rules are changed.

### 3. Search keyboard shortcut

The supplied acceptance criteria require Command/Ctrl-K. The current implementation and automated tests deliberately make search click/tap/tab accessible and disable the shortcut, consistent with the earlier instruction that the site should not use Ctrl-K.

Recommended decision: preserve the current click-only behavior unless that earlier product decision is explicitly reversed. This is the only search acceptance item intentionally in conflict with the current product requirement.

## Workstream 1 — Technical foundation

### Robots policy

`robots.txt` is present, reachable, and references the root and child sitemaps. It explicitly names many search and AI crawlers. Required corrections:

- Add explicit standard and retrieval agents missing from the supplied policy: Googlebot, Bingbot, Claude-SearchBot, Claude-User, and Perplexity-User.
- Reconcile the declared `ai-train=no` signal with crawler-specific `Allow` rules after the author confirms the training stance.
- Treat CCBot and Bytespider according to the confirmed stance and add the requested limitation note about non-compliant bots and browser-signature agents.
- Preserve the `noindex` protection on private/non-public material; do not rely only on crawler directives.

### Sitemap and feed

The root sitemap is currently a static sitemap index. `sitemap-all.xml` lists the 1,042 local canonical pages, but the generator derives `lastmod` from filesystem modification time. That is not necessarily the publication or content-modification date. The Pages workflow does not regenerate the sitemap or RSS feed from the final staged artifact.

The federated production search build reports 42 stale URLs, primarily under retired or moved `Hexstrike-AI-guide` and `ai-vs-defense` paths. A tolerance in the search validator currently permits up to 60 stale sitemap entries; this conflicts with the requirement that sitemaps contain only canonical, reachable URLs.

`feed.xml` is present and valid at a basic XML/RSS level, but it is a checked-in four-item feed rather than a build-generated articles feed. Its completeness and dates can drift from `/articles/`.

Required implementation:

1. Generate sitemap and feed files from the final staged page set.
2. Use explicit `dateModified`/`datePublished` metadata where present, a deterministic Git history date where appropriate, and no misleading date when neither source is authoritative.
3. Validate every emitted URL as indexable, canonical, unique, and in scope.
4. Fail the deployment on stale sitemap URLs rather than accepting a fixed tolerance.

This requires a deployment-workflow change and therefore needs approval before implementation.

### Canonicals

Canonical coverage is strong: every indexable local page has exactly one self-canonical URL. The local article companions currently canonicalize to 1200km.com, not to Medium. Redirect/legacy copies are excluded from indexing. Preserve this behavior and add it to the complete-site validation rather than the current 15-page sample.

### Semantic HTML

The heading hierarchy checker found no H1-to-H3 level gaps, but two H1 defects remain:

- `articles/adversarygraph-v2-self-hosted-ai-cti-platform.html` contains two H1 elements.
- `threat-matrix/index.html` contains multiple H1 elements.

`cv.html` has no `<main>` landmark. Only 225 pages expose both semantic `<header>` and `<footer>` elements; the generated Threat Matrix and documentation families often use a `<main>` and `<nav>` but omit the outer semantic landmarks. Hand-editing hundreds of generated files would be fragile. Fix the relevant generator/template or apply a deterministic staged post-processor, then validate the emitted artifact.

### Performance

Positive findings:

- Pagefind is chunked and loaded on demand.
- Search scripts are deferred/lazy rather than blocking primary content.
- All 872 images have an `alt` attribute.

Open items:

- 711 images lack explicit width and height, which can contribute to layout shift.
- The 472 images without `loading="lazy"` need classification; add lazy loading only below the fold.
- Meaningful alt text must be sampled by page family; attribute presence alone does not prove useful descriptions.
- Google Fonts and other render-path assets require a measured Lighthouse pass before changing loading behavior.
- No repository gate currently proves green Lighthouse Performance, SEO, and Accessibility scores.

Do not declare Core Web Vitals or Lighthouse acceptance complete until measured against the staged production artifact.

## Workstream 2 — Connected structured data

JSON-LD exists on 1,038 of 1,042 indexable pages and all inspected blocks parse as JSON. Missing pages are:

- `ITDR/docs/roadmap/index.html`
- `ITDR/index.html`
- `adversarygraph-docs/index.html`
- `threat-matrix/index.html`

The current vocabulary is broad—Person, TechArticle, ProfilePage, BreadcrumbList, CollectionPage, SoftwareApplication, FAQPage, WebSite, and others are present—but it is not a consistent connected graph. In particular:

- `WebSite` appears on only 6 pages.
- `Organization` appears on only 5 pages.
- `BreadcrumbList` appears on 211 pages.
- `WebPage` appears on only 2 pages.
- Many pages contain standalone Person, ProfilePage, or TechArticle objects without shared `@id` references to a canonical Person, Organization, and WebSite entity.

The current SEO checker verifies parseability and selected required types, not Schema.org relationships, property validity, or external rich-result eligibility. Therefore “zero errors” is not yet proven.

Recommended implementation:

1. Define stable canonical IDs such as `https://1200km.com/#person`, `/#organization`, and `/#website`.
2. Emit a per-page `@graph` with shared entity references, WebPage, and real breadcrumbs.
3. Add only page-type schemas supported by visible content; do not synthesize FAQ content.
4. Preserve honest dates and existing claims. Flag missing author-controlled facts rather than inventing them.
5. Add full-site local relationship tests, then manually validate representative templates in Schema.org Validator and Google Rich Results Test. Record the tested URLs and results.

The existing homepage `SearchAction` is harmless for non-Google consumers, but it no longer enables a Google sitelinks search box. Remove it for a leaner graph or retain it with that limitation documented; it is not a priority feature.

## Workstream 3 — Answer-engine content structure

The site already has strong information architecture, descriptive titles, evidence trails, author identity, and substantial original technical material. A broad automated rewrite would risk weakening the author's voice and introducing claims. Content restructuring should therefore be targeted and reviewed page by page.

Question-style H2 coverage is uneven:

| Area | Pages | H2 elements | Question-style H2 elements |
| --- | ---: | ---: | ---: |
| Root portfolio/hubs | 17 | 120 | 4 |
| Articles | 3 | 46 | 5 |
| AdversaryGraph docs | 82 | 445 | 42 |
| ITDR | 125 | 703 | 54 |
| Threat Matrix | 817 | 3,104 | 0 |

Question wording is a content choice, not a mechanical compliance target. Threat Matrix entity pages are reference records, where canonical entity headings are more useful than forced questions. Prioritize flagship project pages, long-form articles, guides, and practitioner workflows. For each selected section:

- Put the direct answer in the opening two or three sentences.
- Keep claims, dates, measurements, and citations unchanged unless the author supplies updates.
- Use practitioner-language questions only when they improve navigation and extraction.
- Keep one coherent concept per section without fragmenting reference material.
- Normalize ATT&CK IDs, actor aliases, and tool names through the same entity source used by search metadata.

No invented claims or statistics were introduced during this audit.

## Workstream 4 — Internal search

The requested Pagefind foundation is already implemented and should be preserved:

- Pagefind 1.5.2 is installed as a development dependency.
- The deployment builds the index from canonical rendered HTML, not from a hand-maintained document list.
- Navigation and footer boilerplate are excluded during indexing.
- Metadata includes title, description, identifier, aliases, collection, and section.
- Ranking boosts exact identifiers and aliases; automated fixtures cover exact ATT&CK/actor queries.
- The search page retains crawlable collection links and a `<noscript>` fallback.
- The production Pagefind bundle is reachable and the latest build indexed 1,642 documents.

Gaps:

1. Search preprocessing places `data-pagefind-body` on the staged `<body>` and excludes boilerplate with Pagefind selectors. This works, but differs from the requested source-level `<main>` markup. Document this deliberate generated-artifact design instead of duplicating metadata in source and output.
2. The UI disables sub-results, so section-anchor deep links are not proven and are unlikely to be consistently exposed. Enable and style heading-level results, then add browser tests asserting the returned `#fragment` exists.
3. Current filtering exposes a broad section classification, not the requested content-type and topic facets. Add filters only from deterministic page metadata; do not infer unreliable tags.
4. The stale federated sitemap URLs must be resolved upstream or excluded from the source sitemap.
5. Keep search click-only unless the author reverses the previous no-Ctrl-K decision.

No replacement search library or additional search dependency is recommended.

## Workstream 5 — AI-agent files

Both `llms.txt` and `llms-full.txt` already exist and are reachable. `llms.txt` functions as a curated content map and correctly distinguishes the AdversaryGraph v6 release from later RAG/MCP work. Preserve the framing that these files are discovery aids, not ranking guarantees.

Before regeneration, manually verify author-controlled claims in `llms-full.txt`, including publication counts and broad evidence statements. The generator should select canonical, public pages and avoid silently turning navigation or stale metadata into factual claims.

## Prioritized implementation plan

Each phase should be a separate reviewable commit. Generated site families should be corrected at their template/source or by deterministic build transformation, not by mass hand-editing snapshots.

### Phase 1 — Artifact integrity and policy (P0)

- Confirm AI training stance.
- Correct crawler groups and policy comments.
- Make sitemap/feed generation deterministic from the staged artifact.
- Eliminate all stale sitemap URLs and make validation fail on recurrence.
- Add full-site canonical, H1, main-landmark, and JSON-LD-presence checks.

Deployment workflow changes require explicit approval before this phase is implemented.

### Phase 2 — Schema graph (P0/P1)

- Establish stable Person, Organization, and WebSite IDs.
- Generate connected WebPage, breadcrumb, and page-type graphs.
- Cover the four pages currently missing JSON-LD.
- Add local graph validation and record external validator results for representative templates.

### Phase 3 — Semantic and image correctness (P1)

- Fix the two H1 defects and missing `<main>`.
- Improve template-level header/footer/article landmarks where semantically appropriate.
- Add dimensions from actual image metadata.
- Apply lazy loading only to below-the-fold images and verify meaningful alt text by page family.
- Run staged Lighthouse measurements and fix measured regressions.

### Phase 4 — Search completion (P1)

- Keep the existing Pagefind pipeline and ranking model.
- Add heading-level deep links and browser assertions.
- Add deterministic content-type/topic facets where source metadata supports them.
- Preserve click-only access unless the shortcut decision changes.

### Phase 5 — Targeted content and documentation (P2)

- Restructure the highest-value articles, guides, and flagship project pages for answer-first extraction with author review.
- Verify `llms.txt`/`llms-full.txt` claims and generation inputs.
- Add `SEO-NOTES.md` and the complete revalidation checklist.
